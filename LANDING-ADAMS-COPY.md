## SECTION 6 — TRANSACTION LIFECYCLE (stepper/7-step flow)

### Eyebrow
```
WHAT HAPPENS WHEN A USER PRESSES CONFIRM
(A brief guide to seven unremarkable events that together constitute a minor miracle)
```

### Section heading
```
"What happens when a user hits Confirm"
```

### Section sub
```
Not a black box. Not a mystery wrapped in a webhook. 
A seven-step journey you can watch in the admin console in real time, 
which is more than can be said for most things.
```

### Steps — titles and descriptions

**Step 1 — Widget loads**
```
Your app serves one script tag. The widget initialises, fetches live rates, 
and displays them within 200 milliseconds. This is, for reference, 
approximately 300 times faster than most compliance approvals.
```

**Step 2 — User enters amount**
```
The widget shows the send amount, the receive amount, and every fee — 
before the user has confirmed anything. We want to stress this point. 
The fee is shown. In advance. You're welcome.
```

**Step 3 — User presses Confirm**
```
The fee breakdown is shown one more time. The user has now seen the fee 
twice. If they are surprised by the fee at this point, this is beyond 
the reasonable scope of our system to address.
```

**Step 4 — KYC runs**
```
Identity verification happens inside the widget, invisibly to the user 
if they've already passed. First-time users see a 47-second flow. 
This is not a typo. It is 47 seconds. We counted.
```

**Step 5 — Risk assessed**
```
Twelve automated checks: counterparty screening, velocity monitoring, 
pattern analysis. The system is, at this point, briefly smarter than 
any individual working at the company that built it. Manual override 
remains available for the humans.
```

**Step 6 — Transaction executes**
```
Ledger updated. Partner share allocated. Admin console notified. 
All of this before the confirmation screen clears, which is the kind 
of thing that sounds impossible and is, in fact, simply well-engineered.
```

**Step 7 — Partner earns**
```
Revenue posts to the partner portal immediately. Visible to the operations 
team. No CSV. No month-end batch. No spreadsheet named "FINAL_v3_ACTUAL_USE_THIS.xlsx". 
Just the number, in the dashboard, being what it is.
```

### FAQ Items

**Q1:** `Do users see fees before they confirm?`  
**A1:**
```
Yes. Every fee is disclosed at the amount-entry step and shown again 
at confirmation. The user has, by the time they press confirm, seen 
the fee twice. MIFID-aligned. We've had this checked. The check is logged.
```

**Q2:** `Where is identity data stored?`  
**A2:**
```
Inside the KYC widget, which runs on our infrastructure. Your platform 
never touches raw identity documents. We hold only what we need, 
for only as long as regulation requires, and not a byte longer. 
The byte in question is not actually the binding unit here but 
we stand by the spirit of the statement.
```

**Q3:** `Can I export transaction data for regulators?`  
**A3:**
```
One-click export for any date range, in the format regulators expect. 
Not the format that's convenient for the software. Not the format 
that requires additional processing. The format they asked for. 
This distinction, while small, has prevented at least two incidents 
we are not at liberty to describe.
```

**Q4:** `What happens when a transaction is flagged?`  
**A4:**
```
Twelve automated checks run on every transaction. If one flags, 
the transaction enters a manual review queue visible in your admin 
console, with full context: counterparty, velocity, risk score, 
and the specific check that triggered it. Nobody is guessing. 
The information is there. It has always been there.
```

**Q5:** `Is there an audit trail?`  
**A5:**
```
Every action — KYC decision, fee configuration change, transaction 
at every step, manual review outcome — is logged with a timestamp, 
the actor who performed it, and the outcome. Immutable. Exportable. 
Queryable. The kind of audit trail that makes the meeting about 
the audit trail approximately 40 minutes shorter.
```

**Q6:** `Who sees what in the admin?`  
**A6:**
```
Your operations team sees everything. Your partner sees only their own 
data. Your compliance officer gets the export function. No shared 
credentials. No "reply-all" equivalent in the data access model. 
Role-based access configured at setup. If you'd like a seventh question 
answered, we encourage you to book a demo.
```

## COPY NOTES FOR DEVELOPER

1. **Voice consistency:** The jokes are part of the product positioning, not decoration. Keep every line exactly as written, including the parenthetical asides.

2. **Disclaimers:** Anywhere the copy mentions simulated data, simulated companies, or demo money — keep those disclaimers. They're legally sensible and also honest, which is thematically appropriate.

3. **The "47 seconds" fact:** Appears 7 times across the full copy. This is intentional — it's the one concrete, unexpected, specific number that makes the KYC claim credible. Do not vary it.

4. **The comma in "No CSV."** — The period after "CSV" in all instances is deliberate. It's the full stop on a sentence that implies "you've been waiting for us to say this."

5. **Don't add exclamation marks.** Douglas Adams does not use exclamation marks to signal enthusiasm. Enthusiasm is conveyed through precision and absurdity. An exclamation mark here would read as corporate panic.

6. **"mostly harmless"** — If used, always lowercase, always in the context the Guide would use it: as a deliberately extreme understatement. Never as a warning.
