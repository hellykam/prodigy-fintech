## Scope Decisions vs Legacy Risely

| Legacy Risely Feature | Prodigy Decision | Reason |
|---|---|---|
| Rewards admin table | IN — read-only table | Consumer app has Rewards/Referral; admin needs visibility into reward balances |
| Marketing (Referral Programs, Banner, Content) | OUT | Separate product, not in demo |
| Liquidity Management (dedicated section) | IN — simulated demo UI | No real BaaS providers but UI shows simulated provider states with action buttons |
| Cards section | OUT | No card issuing in Prodigy |
| SEPA / Wire Transfer management | OUT | Bank simulator handles internally |
| Compliance → KYT as sidebar item | IN → as Risk Queue | Maps to RiskReview domain model |
| Compliance → KYC as sidebar item | IN → as KYC Queue | Maps to KYCApplicant domain model |
| Settings → Fees | IN → simplified | WidgetConfig has feeProfileId; need Fee Profile view |
| Settings → Users | IN | Maps to admin staff management |
| Settings → Roles | OUT for demo | Hardcoded roles (admin, support, compliance) is enough |
| Activity → Logs | IN → as System Events | Maps to SystemEvent domain model |

## Global UX Rules (apply to every screen)

From `docs/ADMIN_UX_RULES.md` — absolute, no exceptions:

- **Only 3 page types:** Dashboard | Full-width Table | Entity Detail
- **No tabs** on any page, including entity detail pages
- **No modals** — secondary actions use `SidePanel` (pushes content, own URL)
- **Every page has a unique URL.** Every entity has `/id` in URL. Every side panel has `?panel=action-name`
- **Text:** top-left aligned, never truncated, wraps fully
- **Numbers:** top-right aligned
- **Inline icons:** always `display: inline-flex`, never `block`
- **Tables:** full viewport width, no horizontal scroll on desktop, horizontal scroll on mobile
- **Right-click any row or entity link** → opens in new tab (use `<a>` wrappers)
- **All search/filter/sort:** server-side (not limited to current page)
- **CSV export:** on every table and every entity detail page; exports all filtered results
- **Live data:** Transactions, KYC Queue, Risk Queue, System Events tables auto-refresh via WebSocket. Show `LiveIndicator` + "N new items" banner when new rows arrive.
- **Filter state** persisted in URL query params at all times

## 1. Dashboard

**URL:** `/admin`
**Page type:** Dashboard
**Component:** `StatCard`, `DataTable` (recent activity), `LiveIndicator`
**Purpose:** Real-time operational overview. Admin opens this first to see what needs attention.

### 1.1 Stat Cards (top row)

| Card | Value | Source | Delta |
|---|---|---|---|
| Total Customers | count of User records | `GET /api/users?count` | +N this week |
| Pending KYC | count of KYCApplicant where status=pending\|manual_review | `GET /api/kyc?status=pending&count` | — |
| Open Risk Reviews | count of RiskReview where status=open\|escalated | `GET /api/risk-reviews?status=open&count` | — |
| Total Transactions | count of all Transaction records | `GET /api/transactions?count` | +N today |
| Platform Revenue (USD) | sum of PartnerCommission.amount where status=paid | `GET /api/commissions?sum` | — |

**States:** loading skeleton → populated → error (show last known value + error indicator)

### 1.2 Recent System Events Feed

`DataTable` of last 20 SystemEvent records, descending by createdAt.
**Columns:** eventName | entityType | entityId (link to entity) | source → target | status | createdAt
**Live:** WebSocket updates — new events prepend to top with pulse animation.

### 1.3 Quick Links

Below stat cards, before the events table:
- "View KYC Queue (N pending)" → `/admin/kyc-queue`
- "View Risk Queue (N open)" → `/admin/risk-queue`

### 1.4 Acceptance Criteria

- [ ] All 5 stat cards render with real DB data (not hardcoded)
- [ ] KYC Queue stat and link reflect actual pending count
- [ ] Risk Queue stat and link reflect actual open count
- [ ] System Events table updates without page reload when backend emits WS event
- [ ] All links navigate to correct pages
- [ ] Loading skeletons shown while data fetches; no layout shift when data arrives
- [ ] Zero console errors

- Currency (with `CurrencyIcon`)

- Provider

Same structure as Transactions List Table (§4.1) but pre-filtered to this user.
**Endpoint:** `GET /api/transactions?userId=:id`

#### Sub-Table: KYC Applications

| Column | Notes |

- Title: "Restore Customer Access"
- Read-only: Customer ID, Current Status
- Reason (optional textarea)
- `[Activate]` (primary) `[Cancel]`

#### Acceptance Criteria

- [ ] URL contains customer ID; page loads correct customer from DB

### 11.2 Provider Cards

| USD | MoneyAmount | MoneyAmount | MoneyAmount |
| BTC | MoneyAmount | MoneyAmount | — |
| ETH | MoneyAmount | MoneyAmount | — |

- Total balance in USD
- Pending Tx count badge

### 11.3 Demo Simulation Controls

This is what makes Liquidity useful for a demo. A panel of action buttons at the top of the page:

| Button | What it does |
|---|---|
| `[Simulate Low Balance — EUR]` | Sets Bank Simulator EUR available to $500 (triggers warning state on card) |
| `[Simulate Provider Offline]` | Sets Bank Simulator status to `offline` (card turns red) |
| `[Simulate Recovery]` | Resets Bank Simulator to normal state (status → online, balance → normal) |
| `[Simulate High Volume]` | Bumps pending Tx count to 312 (shows high-load state) |
| `[Reset to Defaults]` | Resets all simulated states to seeded defaults |

**Implementation:** Each button calls `POST /api/liquidity/simulate` with an action enum. Backend updates in-memory state (no DB write needed). WebSocket broadcasts the new state to all connected admins. Card updates without page reload.

**Warning states on cards:**
- Available < 10% of normal → card border turns warning color, StatusPill = `degraded`
- Status = `offline` → card border turns error color, StatusPill = `offline`
- Normal → StatusPill = `online`
